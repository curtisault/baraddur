# Governance Check

Evaluate this project's compliance with crates.io publishing policies, the Rust Code of Conduct, and Homebrew governance requirements. Then write `docs/governance.md`.

## Steps

### 1 — Gather project state

Read the following files:
- `Cargo.toml` — check every `[package]` field
- `LICENSE-MIT`, `LICENSE-APACHE`, or any other `LICENSE*` files — verify they exist

Also run:
```
cargo package --list 2>/dev/null
```
to estimate the packaged file list (use this to judge whether the `.crate` is likely to exceed 10 MB).

### 2 — Evaluate crates.io requirements

For each row in the crates.io table below, assign one of:
- ✅ Pass
- ⚠️ Warning (recommended/optional gap, or minor issue)
- ❌ Fail (required field missing or policy violated)
- N/A (not applicable)

| Row | How to evaluate |
|---|---|
| `name` — valid identifier, no impersonation or squatting | Read `[package].name`; verify it is a valid Rust identifier and doesn't impersonate a known person, org, or trademark |
| `description` | `[package].description` must be present and non-empty |
| `license` or `license-file` | One of these must be present in `[package]` |
| License files in repo | For each SPDX identifier in the license expression, a corresponding file must exist (MIT → `LICENSE-MIT`, Apache-2.0 → `LICENSE-APACHE`, etc.) |
| Genuine functionality | Assess whether the project has real, working code — not a placeholder |
| DMCA / trademark / copyright | Check that the crate name and contents don't infringe obvious trademarks; Tolkien-inspired names are acceptable (names aren't copyrightable) |
| Code of Conduct | README and docs must be respectful; no harassing, exclusionary, or offensive language |
| `.crate` file ≤ 10 MB | If `cargo package --list` shows a large binary or media tree, flag ⚠️; otherwise ✅ |
| `repository` | `[package].repository` must be present |
| `readme` field | `[package].readme` should be explicitly set (even if `README.md` is auto-detected) |
| `homepage` | `[package].homepage` should be present |
| `keywords` | `[package].keywords` should be present |
| `categories` | `[package].categories` should be present |

### 3 — Evaluate Homebrew governance requirements

For each row in the Homebrew table below:

- **Code of Conduct** — same content assessment as above
- **Maintainer-role rows** — apply only if the project owner is an active Homebrew maintainer. If not, mark all such rows `N/A` with note: "Not yet a Homebrew maintainer; required if/when contributor role is obtained."
- **Security emergency compliance** — same `N/A` note unless maintainer role is held.

### 4 — Write docs/governance.md

Write `docs/governance.md` using this exact structure, replacing every `<!-- STATUS -->` placeholder with the emoji status and every `<!-- NOTES -->` with a brief inline note (file name, field name, "present" / "absent" / "not applicable"). Keep notes short — one phrase per cell.

```markdown
# Governance Compliance

Last checked: <today's date>

This document tracks compliance with publishing and community standards for the baraddur project.
It is generated by the `/governance-check` skill and should not be edited manually.

---

<populated crates.io table>

---

<populated Homebrew table>

---

_Run `/governance-check` to refresh this file._
```
